Objective

The objective of this assignment is to gain hands-on programming experience with Erlang and to get an understanding of how to implement a simple server.

The goal of the assignment is to implement the backend part of a Kahoot!-like service called Quizmaster!. That is, a Quizmaster! server is a central server where you create new quiz. When you have created a quiz you need to add some questions to it before you can play the quiz. When your quiz is ready, you can open the quiz for players. The players need to join the quiz before they can answer the questions in the quiz. To play a quiz there needs to be exactly one conductor and a number of players.

The play of a quiz is that the conductor makes the questions active in turn, and then the players’ guesses are collected. After each question, each player is awarded points based on the correctness of the answer.

A quiz can be in a number of different states: it can be editable where you can add questions, it can be played when it’s being played, when it’s being played it can either have an active question or it can be between questions.

A player is awarded one point for a correct answer.

API

Implement a module quizmaster with the following API, in the following Q is the process ID of a Quizmaster! server:

  • start() for starting a Quizmaster! server. Returns {ok, Q} on success or {error, Reason} if some error occurred. The quiz starts as being editable.

  • add_question(Q, {Description, Answers}) for adding a question to the quiz managed by Q. Description should be a string; Answers should be a non-empty list of answers, where each answer is either a string or a pair on the form {correct, Text} where Text is a string, the latter form is used for marking which answers are correct (there should be at least one, but there can be multiple correct answers).

    Returns either ok or {error, Reason} if the arguments were on the wrong form or the quiz wasn’t editable.

  • get_questions(Q) for getting the current questions in the quiz. Returns a list of tuples on the form given as the second argument to add_question. The questions should be returned in the order that they were added, that is, the question that was added first should be the head of the list.

    In the (rare?) case where there have been concurrent processes calling add_question there are multiple correct orders because of data races.

  • play(Q) for starting the quiz. Returns {error, no_questions} if no questions have been added. Otherwise returns the atom ok. The process calling this function is then the conductor for the active quiz. When the quiz starts there is no active question and quiz is being played.

    A number of messages can be sent to the conductor; these messages will all have the form Msg, where Msg is one of:

    1. {player_joined, Nick, Active} for informing the conductor that the player Nick has joined, thus bringing the number of players in the quiz up to Active.
    2. {player_left, Nick, Active} (which brings the number of players in the quiz down to Active).

    The purpose of the conductor is to progress the quiz by communicating with the server; like university course examiners, different conductors can have different ideas of how much time each question gets, and how the scores are presented.

  • next(Q) used by the conductor for advancing the quiz to the next question, and making that question active. Returns either {ok, {Description, Answers}} if the process calling the function is the conductor, where Description and Answers are for the next question (and thus have the same types as in add_question); or {error, who_are_you} if the calling process is not the conductor; or {error, has_active_question} if there is already a currently active question.

    When the next function is called, the sever sends a message {next_question, Ref, {Description, Answers}} to all players who are currently joined, where Ref is the value returned by join when the player joined (see below).

  • timesup(Q) used by the conductor for making the current question inactive. Returns either:

    • {ok, Dist, LastQ, Total, Final} with information about the last question, where: Dist is a list of integers giving the distribution of guesses on the different possible answers (thus it has the same length as the number of possible answers for the question); LastQ and Total describe the score after the last question, both are maps mapping player names to points (following the rules laid out in the Objective section), for the last question and for the total standing respectively; and Final is a Boolean that is true if it was the final question of the quiz, and false otherwise. If it was the final question, then the quiz is no longer being played.

      If it was the final question, then the server sends a message {Q, quiz_over} to all players, and considers all players (and the conductor) as left.

    • {error, Reason} if some error occurred. Reason should be no_question_asked if the quiz is active, but there is no active question; it should be nice_try if the calling process is not the conductor.

  • join(Q, Nick) called by a player process to join the quiz using the nickname Nick. Returns {ok, Ref} if the player is considered joined, the Ref should be used for future communication from the player to the Quizmaster! server. Otherwise, if the nickname Nick is taken the function should return {error, is_taken}. The Q should inform the conductor of the new player. The process calling join will receive the next_question messages sent to players when next is called. If one process is responsible for multipler players, it will receive a message per player.

  • leave(Q, Ref) called by a player process to leave the quiz. Returns ok if the Ref corresponds to a known player, or {error, who_are_you} if it does not.

  • guess(Q, Ref, Index) called by a player for making a guess that the correct answer to the current active question is Index. Thus, Index should be larger than or equal to one (because Erlang lists are 1-indexed).

    If there is no active question, or the Index is out of range, then the guess is ignored.

  • Not all functions need be permitted in all Quizmaster! states. For example, it is not specified in the above whether a player may join before the quiz starts.

    We expect you to decide on your own how to handle these cases, explain and argue for them in your report, and verify their implementation in your tests.

    However, in no case may incorrect use of the API functions crash a running Quizmaster! server.

How to get started

  • Start by implementing a client program that starts a Quizmaster! server, adds three questions, and finally asks to get them back. Now you have the first simple test for your server (which will not be able to run until you actually implement the quizmaster module).

    Bonus points if your quiz is about the Advanced Programming course, and double bonus points if the questions contain puns (or other silliness).

  • Then make a skeleton for the quizmaster module with dummy implementations for all functions in the API, and implement the parts of the quizmaster module that correspond to your simple test in the previous point.

  • After that, then implement enough of the quizmaster module so that a conductor can play a quiz without any players (calling next and timesup until there are no more questions), and test your code.

Anticipated Questions

  • Q: Can you add questions to a quiz while the quiz is being played?

    A: No.

  • Q: There is too much to do. Can I pass even if I don’t complete everything?

    A: Yes, if you make a tasteful selection of what to leave out, document it clearly, and thoroughly test the existing parts.

  • Q: What happen if a player sends multiple guesses to a question?

    A: The first received counts, the rest are ignored.

Hand-in Instructions

You should hand in two things:

  1. A short report, report.pdf, explaining the main design of your code, and an assessment of your implementation, including what this assessment is based on.

    In this assignment, it is of particular interest that you discuss how you deal with players who send guesses while there are no active questions.

    Also, if you have thoughts on how the API could be improved, feel free to share them.

  2. A ZIP archive handin.zip, containing one directory src, containing your source code and tests. Your implementation of the quizmaster module (thus in a file called quizmaster.erl) and the code you have used to test your module in other file(s). Your test files should start with test_, so that our automated test can ignore them.

Make sure that you adhere to the types of the API, and test that you do.

To keep your TA happy and win fabulous prizes, follow these simple rules:

  1. The Erlang compiler, with the parameter -Wall, should not yield any errors or warnings for your code.
  2. You should comment your code properly, especially if you doubt its correctness, or if you think there is a more elegant way to write a certain piece of code. A good comment should explain your ideas and provide insight.
  3. Adhere to the restrictions set in the assignment text, and make sure that you export all of the requested API (even if some functions only have a dummy implementation).
  4. Describe clearly what parts of the assignment you have solved.
